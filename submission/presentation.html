핵심 문제

슈퍼센트는 빠르게 실험한다. 그러나 실험의 결과는 조직이 아닌 개인 혹은 팀에만 남는다.

메카닉 실험 결과 가설 검증 마케팅 성과 경쟁사 분석

위 지식들이 개인의 기억과 분산된 문서에만 남아, 다음 기획 사이클에 활용되지 못하고 소멸된다.

주 사용자
게임 기획자

"이 메카닉, 다른 팀에서 테스트해봤나?" — 답을 찾으려면 여러 팀에 일일이 물어봐야 한다. 지식은 각 팀에 분산되어 있지만, 조직 차원에서 빠르게 확인할 수 있는 구조가 없다.

Pain Point
• A팀의 메카닉 실험 결과를 B팀이 알 방법이 없어 동일 실험 반복
• 10개 팀의 경험이 축적되어 있어도 신규 기획자는 처음부터 시작
• 조직 전체의 실험 이력이 문서·슬랙에 파편화되어 있어 AI로 접근 불가
부 사용자
데이터 분석가

지표 데이터와 기획 맥락이 분리되어 통합 인사이트 도출 어려움

마케터

게임 내 메카닉 변화와 마케팅 성과를 연결해 분석하기 어려움

① 문제 정의
이게 진짜 문제인가? — 인터뷰 아티클에서 발견한 것
표면적 시그널 — 문제 없어 보임
바울 (Lead PD) 인터뷰

"혼자 10번의 결과를 보는 것보다, 열 팀의 결과를 함께 보는 게 훨씬 빠르게 성장하는 방법이라고 생각해요. 슈퍼센트는 그게 자연스럽게 일어나는 곳이에요."

→ 얼핏 보면 지식 공유가 이미 잘 되고 있다는 시그널
그렇다면 왜 문제인가?

"자연스럽게"의 실체는 — 바울이라는 시니어 1인의 기억과 경험을 통해 일어난다는 뜻. 바울이 없으면 지식 흐름도 멈춘다.

인터뷰에서 발견한 구조적 문제
사례 1 — 솔 (Game Designer)

"자신만만하게 준비한 코어한 플레이가 처참하게 실패 → 잽싸게 롤백" — 과거 실패 패턴을 검색할 수 없어 반복 실수 발생

사례 2 — 바울 (Lead PD)

"열 팀의 결과를 함께 본다"는 문화적 공유는 존재하지만, 구조화된 검색·축적 시스템 없이 바울의 경험에 의존

사례 3 — 네이선 (Game Designer)

데이터 분석 능력이 핵심이라고 강조하지만, 지표 데이터와 기획 논의가 연결된 통합 저장소 부재 — 인사이트가 개인 문서에 분산

인터뷰가 확인해준 것 — 슈퍼센트는 빠르게 실험한다. 하지만 그 결과가 조직의 자산으로 축적되는 구조가 없다. 이것이 진짜 문제다.

① 문제 정의
지식 단절의 4가지 양상
A
지식 접근성 문제

과거 게임들의 데이터와 논의 결과가 있지만 다음 게임 기획에 활용이 안 됨. 시니어(바울)에게 직접 물어봐야만 알 수 있는 정보 존재.

B
지식 단절 문제

회의에서 나온 인사이트, 가설, 결과가 체계적으로 기록되지 않아 회의 종료 후 소실됨. 같은 논의가 반복됨.

C
팀 간 데이터 사일로

기획팀의 가설, 데이터팀의 지표, 마케팅팀의 캠페인 성과가 서로 연결되지 않아 통합 인사이트 생성 불가.

D
성공/실패 패턴 미활용

과거 게임들의 성공 패턴과 실패 패턴이 조직 자산으로 축적되지 않아 신규 게임 기획 시 반복 실수 발생.

실험은 반복되고, 지식은 사라진다 — 도구의 문제가 아닌 구조의 문제다

B1
자연어 질문 입력

↓ question string
B2
text-embedding-3-small — 질문 벡터화

↓ query_vector [1536-dim]
B3
PostgreSQL(pgvector) — 코사인 유사도 검색
7개 모델 테이블 순회 · CosineDistance(embedding, query_vector) · 각 Top-3
↓ 최대 21개 후보 → 거리 오름차순 정렬 → Top-10
B4
RAG 컨텍스트 구성
Top-10 노드의 type · name · description → system prompt에 삽입
↓ [system: 컨텍스트 노드] + [user: 질문]

↓ answer + referenced_nodes (Top-5)

① 문제 정의
왜 지식은 조직에 남지 못하는가 — 4가지 구조적 단절
A
지식 접근성 문제

과거 게임들의 데이터와 논의 결과가 있지만, 다음 기획에 활용하려면 담당자에게 직접 묻거나 분산된 문서를 뒤져야 한다. 지식에 접근하는 커뮤니케이션 비용이 크다.

B
지식 단절 문제

회의에서 나온 인사이트, 가설, 결과가 체계적으로 기록되지 않아 회의 종료 후 소실됨. 같은 논의가 반복됨.

C
팀 간 데이터 사일로

기획팀의 가설, 데이터팀의 지표, 마케팅팀의 캠페인 성과가 서로 연결되지 않아 통합 인사이트 생성 불가.

D
성공/실패 패턴 미활용

과거 게임들의 성공 패턴과 실패 패턴이 조직 자산으로 축적되지 않아 신규 게임 기획 시 반복 실수 발생.

실험은 반복되고, 지식은 사라진다 — 도구의 문제가 아닌 구조의 문제다

① 문제 정의
구조의 문제에는 구조적 해결이 필요하다
문제의 본질

지식은 만들어지지만 조직에 남지 않는다. 구조가 없기 때문이다.

도구(노션·슬랙)의 문제가 아니라, 지식을 구조화하고 연결하는 시스템이 없는 것이 근본 원인이다.

해결이 충족해야 할 3가지 조건
①
구조화 — 지식을 노드·관계로 축적해 맥락이 보존돼야 한다
②
접근성 — 누구나 자연어로 조직 지식에 접근할 수 있어야 한다
③
연결성 — 팀·게임 간 지식이 연결되어 재활용 가능해야 한다
왜 온톨로지 + RAG인가? — 대안 비교
노션 / Wiki
구조화 없음, AI 검색 불가, 관계 표현 불가
→
온톨로지
노드-엣지로 관계 축적. 메카닉→가설→결과 맥락 보존
슬랙 / 키워드 DB
키워드 매칭만 가능, 의미 파악 불가, 맥락 단절
→
pgvector (RAG)
의미 기반 유사도 검색. "비슷한 메카닉" 자연어 질문 가능
LLM 단독 연결
조직 데이터 없음, 할루시네이션, 일반 지식만 응답
→
LLM + RAG
온톨로지 노드를 컨텍스트로 주입 → 조직 데이터 기반 응답
결론 — 두 기술이 함께 필요한 이유

온톨로지가 없으면 지식이 구조 없이 쌓인다. RAG가 없으면 자연어로 접근할 수 없다. 두 기술의 결합만이 구조적 축적 + 누구나 접근 가능을 동시에 실현한다.

C1
자연어 질문 입력
POST /ask — question 문자열 수신
↓ question string
C2
text-embedding-3-small — 질문 벡터화
gpt_service.embed(question) → 1536-dim query_vector
↓ query_vector [1536-dim]
C3
PostgreSQL(pgvector) — 코사인 유사도 검색
7개 모델 테이블 순회 · CosineDistance(embedding, query_vector) · 각 Top-3
↓ 최대 21개 후보 → 거리 오름차순 정렬 → Top-10
C4
RAG 컨텍스트 구성
Top-10 노드의 type · name · description → system prompt에 삽입
↓ [system: 컨텍스트 노드] + [user: 질문]
C5
gpt-5.4-mini — 컨텍스트 기반 답변 생성
gpt_service.ask_llm(question, context_nodes) → answer 문자열
↓ answer + referenced_nodes (Top-5)
결과
인사이트 응답 + 참조 노드 표시
JSON {answer, referenced_nodes} → 채팅 UI 렌더링

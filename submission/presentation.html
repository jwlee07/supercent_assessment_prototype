해결하려는 문제 & 대상 사용자
핵심 문제

게임 개발 조직에서 반복적으로 쌓이는 지식 — 메카닉 실험 결과, 가설 검증, 마케팅 성과, 경쟁사 분석 — 이 개인의 기억과 분산된 문서에만 존재하며, 다음 게임 기획 시 활용되지 못하는 지식 단절 문제.

대상 사용자
🎯 주 사용자 — 게임 기획자 (PM)

"이런 메카닉, 전에 해봤나? 결과가 어땠나?"를 바로 알고 싶지만, 시니어 한 명의 머릿속에만 있는 상황

부 사용자
📊 데이터 분석가 📣 마케터 ⚙️ 개발자
현재 업무 방식 — 인터뷰 근거
바울 (Lead PD)

메카닉 실험 결과를 기억에서 꺼내 팀에 공유 → 바울 1인 의존도 높음

솔 (Junior PD)

과거 실패 사례를 모르고 같은 방향 시도 → "전에 해봤는데 안 됐어요" 피드백

네이선 (Data Analyst)

지표 데이터와 기획 논의가 연결되지 않아 통합 인사이트 생성 불가

① 문제 정의
주요 불편 사항
A
지식 접근성 문제

과거 게임들의 데이터와 논의 결과가 있지만 다음 게임 기획에 활용이 안 됨. 시니어(바울)에게 직접 물어봐야만 알 수 있는 정보 존재.

B
지식 단절 문제

회의에서 나온 인사이트, 가설, 결과가 체계적으로 기록되지 않아 회의 종료 후 소실됨. 같은 논의가 반복됨.

C
팀 간 데이터 사일로

기획팀의 가설, 데이터팀의 지표, 마케팅팀의 캠페인 성과가 서로 연결되지 않아 통합 인사이트 생성 불가.

D
성공/실패 패턴 미활용

과거 게임들의 성공 패턴과 실패 패턴이 조직 자산으로 축적되지 않아 신규 게임 기획 시 반복 실수 발생.

단순한 도구/프로세스 문제가 아닌, 조직 지식이 개인에게 집중되어 AI로 민주화하지 못한 구조적 문제

② 해결 방법
시나리오별 시스템 흐름도 (개발자 관점)
시나리오 A — 데이터 업로드 → 온톨로지 축적
A1
파일 업로드
HTTP POST /upload_audio — 음성 파일 수신
↓ 음성 파일 (mp3/wav)
A2
CLOVA STT — 음성 → 텍스트
clova_service.transcribe() → transcript 문자열 반환
↓ transcript (자연어 텍스트)
A3
gpt-5.4-mini — 엔티티 추출
gpt_service.extract_entities() → nodes / edges JSON
↓ {nodes: [{type, name, ...}], edges: [...]}
A4
PostgreSQL — 노드 레코드 저장
Mechanic / Hypothesis / Result 등 Django Model → DB row
↓ 저장된 obj (id 확보)
A5
text-embedding-3-small — 벡터 생성 후 같은 행에 저장
gpt_service.embed() → 1536-dim vector → obj.embedding 컬럼 UPDATE
↓ 관계 테이블에 엣지 기록 (GameMechanic 등)
결과
온톨로지 그래프 갱신
get_graph_data() → vis.js 노드/엣지 렌더링
시나리오 B — 자연어 질문 → LLM 인사이트 (RAG)
B1
자연어 질문 입력
POST /ask — question 문자열 수신
↓ question string
B2
text-embedding-3-small — 질문 벡터화
gpt_service.embed(question) → 1536-dim query_vector
↓ query_vector [1536-dim]
B3
PostgreSQL(pgvector) — 코사인 유사도 검색
7개 모델 테이블 순회 · CosineDistance(embedding, query_vector) · 각 Top-3
↓ 최대 21개 후보 → 거리 오름차순 정렬 → Top-10
B4
RAG 컨텍스트 구성
Top-10 노드의 type · name · description → system prompt에 삽입
↓ [system: 컨텍스트 노드] + [user: 질문]
B5
gpt-5.4-mini — 컨텍스트 기반 답변 생성
gpt_service.ask_llm(question, context_nodes) → answer 문자열
↓ answer + referenced_nodes (Top-5)
결과
인사이트 응답 + 참조 노드 표시
JSON {answer, referenced_nodes} → 채팅 UI 렌더링

실제 프로토타입 제작 방식 — Cursor Subagents

단순 코딩이 아닌, Cursor Subagents를 활용한 AI 협업 개발 방법론으로 구현했습니다. 기획자 · 디자이너 · 개발자 3개 Agent가 순차적으로 산출물을 생성하며, 각 Agent는 이전 Agent의 산출물을 참조 문서로 활용합니다.
